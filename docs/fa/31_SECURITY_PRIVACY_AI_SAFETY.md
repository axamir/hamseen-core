# ۳۱ — امنیت، حریم خصوصی و ایمنی AI

## مدل تهدید

| تهدید | کنترل پیشگیرانه | کشف و پاسخ |
|---|---|---|
| مشاهده داده مجتمع دیگر | tenant scope اجباری در API و query | آزمون cross-tenant و هشدار Audit |
| تصاحب حساب | OTP محدود، MFA نقش حساس، نشست کوتاه و ابطال | هشدار ورود غیرعادی و بستن نشست‌ها |
| افزایش سطح دسترسی | RBAC سمت سرور، تأیید دوم | گزارش تغییر نقش و بازبینی دوره‌ای |
| دستکاری مالی | دفتر دوطرفه، idempotency، جداسازی وظایف | تطبیق، before/after و هشدار مغایرت |
| افشای سند/فایل | URL کوتاه‌عمر، اسکن، رمزگذاری | لاگ دانلود و ابطال دسترسی |
| سوءاستفاده API | rate limit، schema validation، WAF | متریک، circuit breaker و پاسخ رخداد |
| دستگاه گمشده | عدم ذخیره راز، قفل نشست، ماسک داده | ابطال همه نشست‌های دستگاه |
| Prompt injection یا نشت AI | منبع allowlist، ابزار محدود، redaction | ثبت تعامل، توقف provider، بازبینی انسانی |
| جعل اعلان/پیام | قالب امضاشده و sender سازمانی | delivery log و گزارش جعل |
| حذف یا تغییر Audit | append-only و hash chaining | بررسی تمامیت و backup مستقل |

## احراز هویت و نشست

- رمز و token هرگز در log یا local storage ناامن ثبت نمی‌شود.
- نقش‌های مدیر، حسابدار، هیئت‌مدیره و سوپرادمین MFA دارند.
- نشست حساس idle timeout و absolute timeout دارد؛ refresh token چرخشی و قابل ابطال است.
- تغییر شماره، بازیابی حساب و اعطای نقش حساس نیازمند تأیید قوی و اعلان است.

## حریم خصوصی و نگهداری

جدول نهایی مدت نگهداری با هیئت‌مدیره و مشاور حقوقی تصویب می‌شود. پیش از آن محیط دمو فقط داده مصنوعی دارد. هر دسته داده هدف، دسترسی، محل ذخیره، مدت پیشنهادی و روش حذف/ناشناس‌سازی دارد. Push/SMS محتوای حساس نمایش نمی‌دهد.

حقوق کاربر: مشاهده اطلاعات خود، درخواست اصلاح، دریافت توضیح، خروج از اعلان اختیاری و درخواست حذف در حدود تعهد قانونی/مالی. درخواست‌ها شماره پیگیری و تصمیم ثبت‌شده دارند.

## مدیریت راز و زیرساخت

- رازها فقط در secret manager محیط؛ `.env.example` بدون مقدار واقعی.
- TLS در انتقال، encryption at rest برای دیتابیس/فایل/backup و رمزگذاری فیلدی داده بسیار حساس.
- محیط local، staging و production جدا؛ داده واقعی وارد local یا Git نمی‌شود.
- dependency و container scan، patch دوره‌ای و حداقل دسترسی سرویس‌ها.

## Backup و تداوم

- backup رمزگذاری‌شده روزانه، نسخه هفتگی مستقل و سیاست نگهداری مصوب.
- آزمون بازیابی دوره‌ای با ثبت RPO/RTO واقعی؛ backup بدون restore test معتبر نیست.
- Runbook قطعی اینترنت، خرابی provider پیام، خرابی دیتابیس و بازگشت به روش اضطراری نگهبانی.

## پاسخ رخداد

چرخه: گزارش → triage → مهار → حفظ شواهد → رفع → بازیابی → اطلاع‌رسانی مجاز → postmortem. شدت‌های `SEV-1` تا `SEV-4`، مسئول Incident Commander، کانال امن و زمان‌های پاسخ پیش از پایلوت تعیین می‌شوند.

## ایمنی AI

- AI فقط از Gateway، با مجوز کاربر، purpose مشخص و حداقل context استفاده می‌کند.
- منابع پاسخ شناسه‌دارند؛ پاسخ بدون داده کافی می‌گوید «اطلاعات کافی نیست».
- اقدامات مالی، دسترسی، اعلان عمومی، حذف، رأی و ورود امنیتی خودکار نیستند.
- Prompt و خروجی حساس با سیاست retention جدا؛ داده برای آموزش provider استفاده نشود مگر قرارداد صریح.
- Kill switch در سطح tenant، قابلیت و provider؛ سقف هزینه و نرخ درخواست.

## Gate پیش از پایلوت

- آزمون RBAC و cross-tenant، scan آسیب‌پذیری، restore rehearsal، بازبینی لاگ، تست آفلاین گیت، tabletop رخداد و تأیید سیاست حریم خصوصی باید ثبت شوند.
